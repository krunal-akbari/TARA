Top Things to Fix First (post-fix)

  1. NEW-1 — the binary-rejection gate counts every high byte (≥128) as printable. _looks_like_text (service.py L252) treats bytes 128–255 as printable, so the only bytes that count against the printable ratio are a handful of low control chars. A null-free binary blob (many JPEGs, UTF-8 mojibake, compressed payloads without an early \x00) sails past the 0.85 gate and is accepted as "text" — re-opening the exact TXT-01 garbage-into-fields hole the fix was meant to close. The binary-reject test only passes because its JPEG sample happens to contain an early \x00.
  2. NEW-4 — RTF strip is naive: hex/unicode/binary escapes leak. RTF_CONTROL_WORD_PATTERN (L49) matches only \word control words. RTF hex escapes \'e9 (accented chars), unicode \uN?, and \bin binary blobs are NOT stripped, so real-world RTF leaves 'e9 fragments and binary bytes in extracted fields. The happy-path test uses only ASCII so it does not catch this.
  3. FIELD-01..05 (unchanged) — heuristics still auto-write to the candidate record and dedupe fingerprint. Phone regex still matches date ranges/IDs; name extraction is still ASCII-only and rejects single-token/ALL-CAPS; email TLD still unbounded. These wrong values still flow into candidate.phone/email/current_company and build_fingerprint (resume_tasks.py L56), risking false cross-candidate dedupe. Out of stated scope for this pass but the highest remaining product risk.
  4. DOCX-03 / DOCX-04 (unchanged) — header/footer text dropped, runs glued with no separator. Names placed in a DOCX header are invisible; tab/line-break-separated contact lines collapse (Emailjane@x.comPhone), defeating the email/phone regexes. Direct accuracy loss for a common resume layout.
  5. DISP-02 (partial) — final empty-text guard still emits one generic message. service.py L296 raises "Could not extract readable text from resume" for every filetype. Typed errors now exist upstream; carry the detected kind into this last guard so support can tell "empty DOCX" from "whitespace TXT".

HighNew NEW-1 · _looks_like_text counts every byte ≥128 as printable — binary rejection is too lax CorrectnessL245–253Confidence: High
Code (current)
def _looks_like_text(data: bytes) -> bool:
    if not data:
        return False
    sample = data[:4096]
    if b"\x00" in sample:
        return sample.startswith((b"\xff\xfe", b"\xfe\xff"))
    allowed_controls = {9, 10, 13}
    printable = sum(1 for byte in sample if byte in allowed_controls or 32 <= byte <= 126 or byte >= 128)
    return printable / len(sample) >= MIN_PRINTABLE_TEXT_RATIO

Why it matters: The clause or byte >= 128 classifies all high bytes (0x80–0xFF) as printable. The only bytes that ever count against the ratio are the low control chars 0–8, 11, 12, 14–31, and 127. A binary file with no early \x00 — and many real binaries (JPEG/PNG bodies, gzip, fonts) are dense with high bytes and sparse with low control bytes — easily clears the 0.85 threshold and is accepted as "text", decoded by latin-1 (which can decode anything), and mined for name/email/phone. This re-opens the very TXT-01 hole the fix was meant to close. The passing test test_binary_upload_is_rejected_not_garbled only works because its JPEG sample contains \x00 at offset 4, hitting the early null-byte branch; drop that null and the test would let the blob through. The intent was almost certainly to allow valid UTF-8 multibyte sequences (which use high bytes) — but the correct way is to attempt a strict decode, not to bless every high byte.

Recommendation (validate via decode + control-char density, not a blanket high-byte pass)
def _looks_like_text(data: bytes) -> bool:
    if not data:
        return False
    sample = data[:4096]
    if b"\x00" in sample:
        return sample.startswith((b"\xff\xfe", b"\xfe\xff"))
    # Prefer a real decode test for the high-byte range:
    try:
        decoded = sample.decode("utf-8")
    except UnicodeDecodeError:
        decoded = sample.decode("latin-1")
    printable = sum(c.isprintable() or c in "\t\n\r" for c in decoded)
    return printable / len(decoded) >= MIN_PRINTABLE_TEXT_RATIO

Using str.isprintable() over decoded chars rejects the C1 control range (0x80–0x9F) and other non-printable Unicode that the raw byte>=128 test waves through, while still accepting legitimate accented UTF-8/Latin-1 text.

Code (current)
def _decode_text(data: bytes) -> str:
    for encoding in ("utf-8-sig", "utf-16", "latin-1"):
        try:
            text = data.decode(encoding)
        except UnicodeDecodeError:
            continue
        if _collapse_spaces(text):
            return _truncate_extracted_text(text)
    raise ResumeParseError("Could not decode resume text")

Why it matters: latin-1 maps every one of the 256 byte values to a character, so data.decode("latin-1") never raises. The only way to reach the final raise is an all-whitespace file. This is fine for the text branch (which is gated by _looks_like_text first) but _extract_rtf_text calls _decode_text with no such gate — so a binary file that happens to start with {\rtf magic (or is uploaded with .rtf/application/rtf) is latin-1-decoded into garbage rather than rejected. Combine with NEW-1 and the binary-reject guarantee is weaker than the report claims. Gate RTF input through _looks_like_text too, or have _decode_text accept a strict-only flag.

MediumNew NEW-4 · RTF strip misses hex (\'hh), unicode (\uN) and binary (\bin) escapes CorrectnessL49, L267–272Confidence: High
Code (current)
RTF_CONTROL_WORD_PATTERN = re.compile(r"\\[a-zA-Z]+-?\d* ?")
# ...
def _extract_rtf_text(data: bytes) -> str:
    text = _decode_text(data)
    text = text.replace("\\par", "\n").replace("\\line", "\n")
    text = RTF_CONTROL_WORD_PATTERN.sub("", text)
    text = text.replace("{", " ").replace("}", " ").replace("\\", " ")

Why it matters: The control-word regex only matches \word-style words. Three common RTF constructs slip through: (1) hex escapes \'e9 — used for every accented character in ansi RTF — the \' is not [a-zA-Z], so after the final replace("\\"," ") you are left with stray 'e9 tokens littering the text and corrupting names like "José" → "Jos 'e9". (2) Unicode escapes \u233? leave the trailing fallback char. (3) \bin<N> binary blobs and embedded objects are not skipped — their raw bytes leak into output. The happy-path test (test_rtf_control_words_do_not_leak_into_extracted_fields) only uses 7-bit ASCII, so none of this is exercised. For an international ATS this directly corrupts the name/company fields it is supposed to clean. Recommendation: use a real RTF-to-text routine (e.g. striprtf) or extend the stripper to decode \'hh and \uN and drop \bin runs.

Overall Verdict

The developer hit the goals: all three Critical findings are genuinely closed — DOCX now parses with defusedxml (dependency confirmed in pyproject.toml) behind real zip-bomb size/ratio caps, the dispatcher routes by magic-byte sniff first, the Celery task has a soft_time_limit with typed timeout/input/infra handlers, and MinerU is verifiably gone. Per-filetype and adversarial tests now exist. The single real new defect is NEW-1: _looks_like_text blesses every byte ≥128 as printable, so a null-free binary can still slip through the text gate and be mined as garbage — partially undermining the TXT-01 fix (the binary-reject test only passes by accident of an early null byte). With NEW-1 tightened, this is production-safe; everything else still open is pre-existing precision/robustness (field heuristics, DOCX run handling, RTF escape handling) that was outside the fix scope. I would approve with nits — fix NEW-1 before relying on the binary-rejection guarantee, and schedule the FIELD-* precision work since those values auto-write to candidate records and the dedupe fingerprint.
